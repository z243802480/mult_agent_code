# 多智能体自主开发系统 - 项目目标

## 1. 愿景

构建一套本地优先的自主研发工作台。用户只需要输入一个较小的系统目标，系统就能通过受控的多智能体循环，持续把目标转化为可使用的软件、研究产物、报告或交互式系统。

这个系统不是智能体聊天室，而是一套智能体运行时。它要把模型调用转化为可验证的工作产物：计划、任务、代码补丁、实验、UI 输出、报告、测试、评审和可复用记忆。

## 2. 核心目标

给定用户的一段简短目标，平台应该能够：

1. 将目标澄清为结构化规格。
2. 根据目标、领域和使用场景推断缺失但合理的需求。
3. 当用户目标描述不足时，自动调研同类工具、常见工作流和标准功能集。
4. 判断是否需要调研、实现、UI、报告、实验，或者它们的组合。
5. 将目标拆解为可追踪任务。
6. 将任务分配给专门角色的智能体。
7. 让智能体在隔离工作区中工作。
8. 产出代码、文档、UI、报告和实验结果。
9. 通过测试、评估、截图和验收条件验证产物。
10. 自动修复失败。
11. 保留成功变更，回滚失败尝试。
12. 将有价值的知识沉淀为长期记忆。

## 3. 自迭代与自主需求扩展

系统不能表现得像一个“一步指令执行器”。它应该把用户的一句简短目标视为一个完整产品、研究或工程任务的种子。

当用户给出较笼统的目标时，系统应该自主完成：

1. 深入理解目标领域，识别该类工具或系统通常应具备的能力。
2. 调研常见工具、工作流、边界场景、用户期待和质量标准。
3. 将初始目标扩展为合理的产品需求集。
4. 区分必需功能、应该具备的功能和可选功能。
5. 构建第一个可用版本。
6. 评估结果只是“能跑”，还是确实“有用、好用”。
7. 针对缺失能力、糟糕体验、薄弱测试或不完整文档创建后续任务。
8. 持续迭代，直到系统达到可用且相对好用的基线，或触及预算和安全边界。

示例：

如果用户说“做一个密码测试工具”，系统不应该只做一个输入框和评分。合格的 ProductAgent 和 ResearchAgent 应该发现常见工具可能包含密码强度评分、熵估算、常见弱密码检测、在允许时的泄露列表检查、哈希示例、编码和解码辅助、随机密码生成、密码短语生成、策略检查、本地隐私保证和清晰的安全提示。系统随后应该根据用户场景判断哪些功能合适，实现一个连贯的功能子集，并评估最终工具是否真的可用。

这种自迭代由模型判断、调研证据、产品启发式规则、自动检查和智能体评审共同驱动。目标不是无限追求完美，而是在合理停止点交付一个可用、连贯、有文档、有测试，并且明显好于字面一步实现的结果。

## 4. 产品定位

本项目是一套自进化研发与软件构建工作台。

它吸收以下方向的思想：

- 智能体编程系统，例如 Claude Code、OpenCode、OpenHands 和 SWE-agent。
- 自主实验循环，例如 Karpathy 的 autoresearch。
- 自动化科研系统，例如 AutoResearchClaw、ARI 和 AI Scientist。
- 具备检索、记忆和结构化项目上下文的本地知识库系统。

系统可以学习公开的架构思想，但不能复用泄露或专有源码。

## 5. 主要使用场景

### 5.1 构建小型软件工具

示例：

“构建一个本地 Markdown 知识库系统，支持语义搜索和问答。”

期望输出：

- 可运行的本地应用。
- 导入和索引流程。
- 搜索和问答界面。
- 测试。
- README。
- 最终实现报告。

### 5.2 自主扩展模糊产品目标

示例：

“做一个密码测试工具。”

期望输出：

- 对常见密码、安全、编码类工具进行产品调研。
- 扩展出必需功能和可选功能需求。
- 实现本地优先且隐私安全的工具。
- 在合适时提供强度评分、策略检查、生成辅助，以及部分编码或哈希工具。
- 清晰说明工具能证明什么、不能证明什么。
- 在最终报告前进行可用性评审和后续迭代。

### 5.3 构建创意驱动的小系统

示例：

“想几个有用的个人自动化工具，并实现其中最好的一个。”

期望输出：

- 创意列表和选择理由。
- 产品规格。
- 可运行原型。
- 根据场景选择 UI 或 CLI。

### 5.4 调研并复现想法

示例：

“调研本地知识库的混合检索方法，并实现最合适的方案。”

期望输出：

- 文献和开源项目调研。
- 结构化观点与证据。
- 实现计划。
- 实验循环。
- 基准结果。
- 保留或丢弃的补丁。

### 5.5 生成报告或知识产物

示例：

“分析一批笔记文件，并生成一份战略 PDF 报告。”

期望输出：

- 数据导入。
- 分析。
- PDF 或 Web 报告。
- 可追踪的来源引用。

### 5.6 改进已有代码库

示例：

“让这个项目更容易使用，并修复明显的问题。”

期望输出：

- 仓库扫描。
- 改进计划。
- 补丁。
- 测试。
- 评审总结。

## 6. 成功标准

MVP 成功的标准：

1. 接收自然语言目标。
2. 生成结构化目标规格和任务计划。
3. 至少能将简单的欠规格目标扩展为合理的验收条件。
4. 至少使用一个编码智能体编辑文件。
5. 运行验证命令。
6. 检测失败并尝试自动修复。
7. 将每次尝试记录到实验日志。
8. 保留成功补丁并丢弃失败补丁。
9. 生成最终报告，说明改了什么、通过了什么、失败了什么、还剩什么。

完整系统成功的标准：

1. 多个智能体能够在隔离工作区并发运行。
2. 调研结果能够转化为实现任务。
3. 能自主将模糊目标扩展为连贯的产品需求和实现计划。
4. 能构建合适的体验产物，例如 Web 应用、CLI 工具、PDF 报告或仪表盘。
5. 跨项目维护长期记忆。
6. 不只评估最终结果，也评估智能体执行轨迹。
7. 持续自迭代，直到产物可用、连贯且相对好用，而不是仅仅“技术上存在”。
8. 通过可复用工作流、技能、命令和评估持续改进。

## 7. 非目标

早期版本不应尝试：

- 取代完整工程团队。
- 在没有策略控制的情况下运行无限制 shell。
- 在智能体运行时稳定前构建庞大的无代码平台。
- 依赖单一专有模型提供商。
- 在核心循环跑通前优先打磨漂亮仪表盘。
- 让大量智能体在没有任务、产物和评估约束的情况下自由聊天。
- 在产物已经达到约定的可用基线后继续无限追求完美。

## 8. 指导原则

1. 先运行时，后智能体。
2. 每次模型调用都应该产出或改进某个产物。
3. 每次变更都应该可验证。
4. 失败尝试是正常路径，必须记录。
5. 保留或丢弃应该基于冻结的评估器。
6. 工具设计和提示词设计一样重要。
7. 上下文应该分层检索，而不是整体倾倒。
8. 多智能体自治需要工作区隔离和合并纪律。
9. UI 和输出形式应该由任务需要决定，而不是习惯决定。
10. 调研必须变成可执行假设，而不是被动摘要。
11. 一个模糊但合理的目标，应该被扩展成完整且有用的结果。
12. 系统不应该等待用户把每个显而易见的产品细节都说出来。
