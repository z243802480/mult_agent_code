# 多智能体自主开发系统 - 需求文档

## 1. 概述

本文档定义一套本地优先的多智能体自主开发系统的功能需求和非功能需求。

系统接收用户目标，进行规划，按需调研，实现软件或报告，验证结果，修复失败，并记录可复用记忆。

## 2. 用户角色

### 2.1 人类所有者

主要用户，负责提供目标、设置预算、审查关键决策，并接收最终产物。

### 2.2 智能体运行时

受控执行环境，负责管理上下文、工具、工作区、权限、记忆、日志和评估。

### 2.3 专门智能体

在运行时约束下工作的角色化模型工作者。

初始角色：

- GoalSpecAgent
- PlannerAgent
- ArchitectAgent
- ResearchAgent
- CoderAgent
- UIExperienceAgent
- TesterAgent
- ReviewerAgent
- AutoCorrectionAgent
- MemoryAgent
- ReleaseAgent

## 3. 功能需求

### 3.1 目标输入

系统必须接收自然语言目标，并将其转换为结构化的 `GoalSpec`。

`GoalSpec` 必须包含：

- 原始用户目标。
- 假设。
- 约束。
- 目标输出形式。
- 完成定义。
- 验证策略。
- 预算限制。

### 3.2 规划

系统必须将目标拆解为里程碑和任务。

每个任务必须包含：

- ID。
- 标题。
- 描述。
- 所属角色。
- 依赖关系。
- 验收条件。
- 允许使用的工具。
- 预期产物。
- 状态。

任务状态：

- `backlog`
- `ready`
- `in_progress`
- `reviewing`
- `testing`
- `blocked`
- `done`
- `discarded`

### 3.3 智能体运行时

运行时必须提供：

- 上下文管理器。
- 工具注册表。
- 权限管理器。
- 决策管理器。
- 工作区管理器。
- 事件日志。
- 评估运行器。
- 记忆存储。
- 预算控制器。
- 恢复和回滚引擎。

### 3.4 上下文管理

系统必须维护分层上下文：

- 全局指令。
- 项目指令。
- 当前任务上下文。
- 检索得到的代码上下文。
- 检索得到的调研上下文。
- 最近执行轨迹摘要。
- 相关记忆。

系统应避免注入不必要的完整历史上下文。

系统必须支持面向长任务的上下文压缩。

上下文压缩必须保留：

- 当前目标和完成定义。
- 已接受的用户决策。
- 活跃任务及任务状态。
- 修改过的文件及修改原因。
- 已运行的测试和验证命令。
- 重要失败和修复尝试。
- 未解决风险、阻塞点和下一步行动。
- 调研观点和已接受的实现假设。

压缩可以由以下方式触发：

- 用户通过 `/compact` 等命令手动触发。
- 当上下文使用量超过可配置阈值时自动触发。
- 在阶段边界触发，例如调研后、实现后或评审前。
- 在将工作交接给另一个智能体前触发。

压缩应支持可选的聚焦指令，例如“保留 API 决策”或“重点保留 UI 反馈”。

### 3.5 工具系统

运行时必须暴露结构化工具，而不是只依赖原始 shell 访问。

必需工具类别：

- 文件读写工具。
- 代码搜索工具。
- 补丁应用工具。
- 测试执行工具。
- Lint 和类型检查工具。
- Web 或文档调研工具。
- Git 和 worktree 工具。
- 浏览器和截图工具。
- 报告生成工具。
- 记忆读写工具。

每次工具调用都必须记录：

- 智能体 ID。
- 任务 ID。
- 工具名称。
- 输入摘要。
- 输出摘要。
- 成功或失败。
- 成本和耗时，如果可用。

### 3.6 命令与工作流系统

系统必须支持用于常见智能体工作流的可复用命令。

命令不是简单的文本快捷方式，而是具名工作流，包含：

- 描述。
- 参数。
- 允许使用的工具。
- 需要的智能体角色。
- 预期产物。
- 安全或审批规则。
- 可选模板或脚本。

初始命令应包括：

- `/init`：初始化 agent-ready 项目工作区，并创建根项目指导文件。
- `/plan`：澄清目标、假设、里程碑和任务。
- `/brainstorm`：生成、比较并排序产品方向或实现想法。
- `/research`：进行结构化调研，并将发现转化为假设或任务。
- `/compact`：压缩上下文，同时保留关键状态。
- `/decide`：创建面向用户的决策点。
- `/review`：评审代码、UX、测试或架构。
- `/debug`：分析失败证据并提出修复。
- `/handoff`：为另一个智能体或未来运行创建机器可读的续接包。

`/brainstorm` 命令必须同时支持发散和收敛：

1. 生成多个想法或解决路径。
2. 聚类相似想法。
3. 按用户目标、可行性、成本、风险和预期价值评分。
4. 选择推荐方向。
5. 从最佳候选中创建任务、实验或决策点。

命令应允许用户调用，并在安全时允许智能体自行调用。

### 3.7 项目初始化与根指导文件

系统必须提供初始化工作流，将一个仓库或空目录准备成适合智能体工作的空间。

初始化工作流应该：

- 判断当前工作区是已有项目、空项目，还是仅用于规划的文档工作区。
- 创建或更新智能体行动前必须可靠读取的根指导文件。
- 记录项目目的、当前假设、约束和非目标。
- 检测技术栈、包管理器、测试命令、构建命令和格式化命令。
- 构建初始项目地图，记录重要目录、入口文件、配置文件和文档。
- 记录安全边界，例如哪些文件未经批准不能触碰。
- 在可发现时记录代码约定和 UI/设计约定。
- 创建默认验证配置。
- 创建默认决策颗粒度配置。
- 创建初始上下文快照。

推荐根文件：

- `AGENTS.md`：面向所有智能体的项目级指令。
- `.agent/project.json`：结构化项目元数据。
- `.agent/context/root_snapshot.json`：初始机器可读上下文。
- `.agent/tasks/backlog.json`：初始任务看板。
- `.agent/runs/`：运行日志和产物。
- `.agent/memory/`：本地项目记忆。

`/init` 必须可以安全重复运行。重复运行时，应更新检测到的元数据，但不能覆盖用户手写的指导内容，除非用户明确要求。

根指导文件必须被所有智能体视为高优先级项目上下文。

### 3.8 权限

系统必须执行基于角色的权限控制。

示例：

- ResearchAgent 可以访问 Web 调研工具，但不能合并代码。
- ReviewerAgent 可以检查 diff，但不应直接修改实现文件。
- CoderAgent 只能编辑分配给自己的工作区文件。
- ReleaseAgent 可以打包产物，但不能在没有明确批准的情况下部署。

高风险操作必须被阻断或要求批准：

- 删除大量文件。
- 读取密钥。
- 修改环境文件。
- 安装全局包。
- 推送到远程仓库。
- 部署到生产环境。

### 3.9 人类决策交互

系统必须支持在重要决策点上进行交互式的人类操盘。

系统不应该因为每个小实现细节都打断用户。常规选择应自主继续，但当某个大的分支决策会实质影响最终结果时，系统应该升级给用户决定。

决策点可以包括：

- 在明显不同的产品方向之间选择。
- 选择输出媒介，例如 Web 应用、CLI、PDF 报告、桌面应用或 API 服务。
- 当技术栈取舍有实质影响时选择技术栈。
- 将范围扩展到原始目标之外。
- 放弃或延期某个预期功能。
- 消耗较大的额外预算。
- 使用外部网络服务或敏感数据。
- 执行不可逆或高风险变更。
- 在多个有调研依据的实现方案之间选择。

每个面向用户的决策请求必须包含：

- 简洁的问题。
- 推荐选项。
- 尽可能提供 2-4 个具体选择。
- 每个选择的取舍。
- 用户不响应时的默认行为。
- 对预算、范围、风险和输出质量的影响。

决策颗粒度必须可配置。

建议模式：

- `autopilot`：只在安全关键或不可逆决策上询问。
- `balanced`：在重大产品、架构、预算或隐私决策上询问。
- `collaborative`：更频繁地询问产品和 UX 选择。
- `manual`：重要变更或范围扩展前都先询问。

系统必须记录每次决策，并将其作为项目记忆使用。

### 3.10 工作区隔离

系统必须支持智能体隔离工作区。

推荐机制：

- Git worktree。
- 临时分支。
- 需要更高隔离时使用容器。

在并发模式下，智能体应提交补丁，而不是直接修改主工作区。

### 3.11 调研自动化

系统必须支持自动化调研循环。

调研输出必须包含：

- 来源。
- 观点。
- 证据。
- 不确定性。
- 候选想法。
- 实现假设。
- 实验计划。

调研在有价值时应创建实现任务。

### 3.12 自动实验循环

系统必须支持受 autoresearch 启发的保留/丢弃循环。

每个实验必须定义：

- 基线。
- 候选变更。
- 冻结评估器。
- 运行命令。
- 变更前指标。
- 变更后指标。
- 决策。
- 理由。

成功实验被保留。失败实验被回滚或归档。

### 3.13 实现

编码智能体必须：

- 编辑前阅读相关上下文。
- 做范围受控的修改。
- 优先遵循现有项目约定。
- 在风险需要时添加测试。
- 产出补丁摘要。

### 3.14 UI 和体验输出

系统必须包含 UI/Experience 智能体，用于决定最合适的输出形态。

可能的输出形态：

- Web 应用。
- 桌面应用。
- CLI。
- TUI。
- PDF 报告。
- Markdown 知识库。
- 仪表盘。
- 浏览器扩展。
- API 服务。

UI/Experience 智能体必须产出：

- 输出媒介建议。
- 交互模型。
- 主要页面或章节。
- 视觉风格方向。
- 可用性验收条件。

### 3.15 验证

系统必须运行合适的检查：

- 单元测试。
- 集成测试。
- Lint。
- 类型检查。
- 构建。
- 冒烟测试。
- UI 截图检查。
- 基准或评估脚本。
- 报告完整性检查。

验证策略应根据目标选择。

### 3.16 自动纠错

当验证失败时，系统必须：

1. 捕获失败证据。
2. 总结日志。
3. 生成假设。
4. 尝试最小修复。
5. 重新运行验证。
6. 保留成功修复。
7. 在超过重试限制后回滚失败尝试。

### 3.17 评审

系统必须在合并前评审补丁。

评审应检查：

- 正确性。
- 回归风险。
- 安全问题。
- 范围膨胀。
- 测试覆盖。
- 可维护性。
- 适用时的 UX 问题。

### 3.18 记忆

系统必须存储可复用记忆：

- 用户偏好。
- 项目决策。
- 架构笔记。
- 成功模式。
- 失败实验经验。
- 工具知识。
- 调研摘要。

记忆检索必须基于相关性，并限定在当前任务范围内。

### 3.19 最终报告

每次运行结束时，系统必须生成最终报告：

- 目标。
- 完成的任务。
- 生成的产物。
- 运行过的测试和评估。
- 指标。
- 保留的变更。
- 丢弃的尝试。
- 剩余风险。
- 建议的后续步骤。

## 4. 非功能需求

### 4.1 本地优先

系统默认应在本地运行，并默认将项目数据存储在本地，除非另有配置。

### 4.2 模型无关

系统应支持 OpenAI-compatible 模型 API，以及智谱、MiniMax、DeepSeek、OpenRouter 和本地模型等提供商。

### 4.3 可观测性

每次运行都必须能通过日志、任务状态、工具调用、模型调用、补丁和评估结果进行追踪。

### 4.4 成本控制

系统必须支持：

- token 预算。
- 时间预算。
- 实验次数限制。
- 每个智能体的预算。
- 停止条件。

### 4.5 可靠性

系统必须能容忍：

- 模型错误。
- 工具失败。
- 坏补丁。
- 测试失败。
- 调研时的网络失败。

### 4.6 可扩展性

系统应支持添加：

- 新智能体。
- 新工具。
- 新技能。
- 新评估。
- 新输出形式。
- 新模型提供商。

## 5. MVP 范围

MVP 应包含：

1. CLI 目标输入。
2. 结构化目标规格生成。
3. 任务看板。
4. 单一隔离工作区。
5. Planner、Coder、Tester、Reviewer、AutoCorrection 和 Reporter 角色。
6. 包含文件、搜索、补丁、shell 和测试工具的工具注册表。
7. 基础事件日志。
8. 带可配置决策颗粒度的基础决策管理器。
9. 包含 `/init`、`/plan`、`/brainstorm`、`/compact`、`/review` 和 `/handoff` 的基础命令系统。
10. 基础保留/丢弃循环。
11. 最终报告生成。

MVP 可以暂缓：

- 完整 Web 仪表盘。
- 大规模并发智能体。
- 完整论文导入。
- 高级向量记忆。
- 生产部署自动化。

## 6. 未来范围

未来版本可以增加：

- 多智能体并行执行。
- 论文导入和引用跟踪。
- 智能体运行可视化仪表盘。
- PDF 报告生成。
- 基于浏览器的 UI 检查。
- 长期向量记忆。
- 分布式实验运行器。
- 插件和技能市场。
- 人类审批工作流。
