# 多智能体自主开发系统 - 任务拆解指南

## 1. 文档目的

本文档定义 PlannerAgent 如何把目标拆成合适颗粒度的任务。

好的任务拆解应该做到：

- 不太大，大到无法一次实现和验收。
- 不太小，小到调度成本高于实现价值。
- 每个任务都有明确产物。
- 每个任务都有可验证验收条件。
- 任务之间依赖清晰。
- 能支持多 agent 协作和后续自动评估。

## 2. 任务拆解原则

### 2.1 以可验收产物为边界

一个任务应该对应一个可检查产物。

好例子：

```text
实现密码强度评分模块，并添加单元测试。
```

坏例子：

```text
优化代码。
```

### 2.2 以用户价值或工程闭环为单位

任务应该能让系统向可用状态前进。

好例子：

```text
实现文件夹导入 Markdown，并生成 Document 列表。
```

坏例子：

```text
创建一个空的 importer.py。
```

### 2.3 单任务建议大小

建议一个任务满足：

```text
预计实现时间：15-90 分钟
文件改动：1-5 个文件
验收条件：2-6 条
依赖数量：0-3 个
输出产物：1-4 个
```

超过这个范围，应考虑拆分。

### 2.4 避免任务过碎

以下任务通常太碎：

- 创建空目录。
- 添加单个常量。
- 修改单个文案。
- 写一个只有一行的函数。

除非它们是风险隔离、迁移步骤或独立验证点。

### 2.5 避免任务过大

以下任务通常太大：

- 实现完整知识库系统。
- 做完整 UI。
- 接入所有模型提供商。
- 完成所有测试。

应该拆成导入、索引、搜索、问答、UI、测试、文档等任务。

## 3. 标准任务结构

```json
{
  "task_id": "task-0001",
  "title": "实现密码强度评分模块",
  "description": "根据长度、字符多样性和常见弱密码规则计算评分，并返回解释",
  "role": "CoderAgent",
  "priority": "high",
  "depends_on": [],
  "acceptance": [
    "输入密码后返回 0-100 分",
    "返回至少 2 条可解释原因",
    "包含常见弱密码检测",
    "包含单元测试"
  ],
  "expected_artifacts": [
    "src/scoring.ts",
    "tests/scoring.test.ts"
  ],
  "verification": [
    "npm test"
  ]
}
```

## 4. 任务类型

### 4.1 Research Task

用于调研和证据收集。

输出：

- 来源。
- 观点。
- 证据。
- 实现假设。
- 决策点或实现任务。

验收：

- 至少 3 条相关发现，除非禁用联网。
- 观点区分来源和模型推断。
- 至少转化出 1 个后续任务或约束。

### 4.2 Product Task

用于需求扩展和产品判断。

输出：

- must/should/could 需求。
- 用户场景。
- 非目标。
- 可用性标准。

验收：

- 明确目标用户。
- 明确完成定义。
- 识别重大决策点。

### 4.3 Architecture Task

用于技术方案和模块边界。

输出：

- 技术栈。
- 模块图。
- 数据流。
- 风险和取舍。

验收：

- 不引入不必要复杂度。
- 给出可实现的模块边界。
- 说明主要取舍。

### 4.4 Implementation Task

用于代码实现。

输出：

- 代码文件。
- 测试文件。
- 实现说明。

验收：

- 功能可运行。
- 测试通过或失败原因明确。
- 改动范围受控。

### 4.5 UI/Experience Task

用于界面、报告或交互产物。

输出：

- 输出形态建议。
- 页面结构或报告结构。
- UI 实现。
- 可用性验收。

验收：

- 交互形态匹配任务。
- 核心流程能完成。
- 文案不误导用户。

### 4.6 Verification Task

用于测试和评估。

输出：

- 测试计划。
- 测试代码。
- 运行结果。
- eval report。

验收：

- 覆盖关键需求。
- 失败可复现。
- 报告明确。

### 4.7 Repair Task

用于失败修复。

输出：

- 失败摘要。
- 修复假设。
- 最小补丁。
- 重新验证结果。

验收：

- 修复目标明确。
- 不是大范围重写。
- 指标改善或明确失败。

### 4.8 Documentation Task

用于 README、报告、使用说明。

输出：

- README。
- final_report。
- API 或命令说明。

验收：

- 用户能运行。
- 说明限制。
- 说明验证结果。

## 5. 任务拆解流程

```text
1. 读取 GoalSpec。
2. 识别完成定义。
3. 识别必须产物。
4. 识别验证策略。
5. 按能力链路拆分任务。
6. 标记依赖。
7. 为每个任务写验收条件。
8. 检查任务大小。
9. 检查是否有缺失角色。
10. 生成任务计划。
```

## 6. 能力链路拆解法

适合软件工具：

```text
输入
  -> 解析/导入
  -> 核心处理
  -> 存储/状态
  -> 输出/展示
  -> 验证
  -> 文档
```

示例：Markdown 知识库

```text
导入 Markdown
  -> 文档切分
  -> 建立索引
  -> 搜索
  -> 问答
  -> UI
  -> 测试
  -> README
```

## 7. 风险优先拆解法

适合不确定性高的任务。

先拆：

- 技术可行性验证。
- 第三方依赖验证。
- 性能验证。
- 隐私和安全验证。
- UX 输出形态验证。

不要一开始就铺开所有功能。

## 8. 垂直切片拆解法

适合 MVP。

优先做一条完整闭环：

```text
最小输入
  -> 最小处理
  -> 最小输出
  -> 最小测试
  -> 最小文档
```

然后再扩展功能。

示例：密码测试工具第一切片

```text
输入密码
  -> 计算基础评分
  -> 展示评分和原因
  -> 单元测试
  -> README 运行说明
```

## 9. 任务质量评分

每个任务生成后应打分：

```text
clarity_score：描述是否清晰
testability_score：是否可验收
size_score：大小是否合适
dependency_score：依赖是否清楚
artifact_score：产物是否明确
risk_score：风险是否可控
```

建议阈值：

```text
clarity_score >= 0.75
testability_score >= 0.75
size_score >= 0.70
artifact_score >= 0.75
```

低于阈值时，PlannerAgent 应重写任务。

## 10. 拆解反模式

### 10.1 只有动词没有产物

坏：

```text
优化搜索。
```

好：

```text
实现 BM25 + 向量混合搜索，并用固定查询集评估命中率。
```

### 10.2 验收条件不可验证

坏：

```text
界面好看。
```

好：

```text
在 1366x768 下核心按钮不重叠，搜索结果列表可滚动，主要流程可完成。
```

### 10.3 一次任务跨太多层

坏：

```text
实现完整知识库，包括导入、索引、搜索、问答和 UI。
```

好：

```text
实现 Markdown 导入器。
实现基础搜索索引。
实现搜索 API。
实现搜索 UI。
```

### 10.4 没有风险隔离

坏：

```text
接入在线泄露密码 API。
```

好：

```text
创建关于泄露检测方案的 DecisionPoint，比较不接入、本地导入和在线 API。
```

## 11. PlannerAgent 输出要求

PlannerAgent 输出任务计划时必须包含：

- 任务列表。
- 依赖关系。
- 阶段划分。
- 每个任务验收条件。
- 每个任务预期产物。
- 每个任务建议角色。
- 重大决策点。
- 风险任务标记。
- MVP 任务和后续任务区分。

## 12. 推荐任务数量

根据目标复杂度：

```text
简单工具：5-10 个任务
中等系统：10-25 个任务
复杂系统：25-60 个任务，必须分阶段
研究复现：调研 3-8 个任务，实验 3-10 个任务
```

MVP 单轮执行建议：

```text
active_tasks <= 8
parallel_tasks <= 3
```

## 13. 任务拆解和成本控制

任务拆解会影响成本。

过大任务：

- 容易失败。
- 修复成本高。
- 上下文复杂。

过小任务：

- 调度成本高。
- 模型调用浪费。
- 报告噪声大。

成本友好的任务应该：

- 有清晰边界。
- 能一次完成。
- 失败时容易定位。
- 产物可复用。

## 14. 任务拆解验收

一个任务计划合格的标准：

- 每个 must 需求至少被一个任务覆盖。
- 每个任务都有验收条件。
- 每个任务都有预期产物。
- 高风险选择有 DecisionPoint。
- 第一阶段任务能形成可运行闭环。
- 暂缓需求进入后续阶段，而不是消失。
