# 多智能体自主开发系统 - 产品规格

## 1. 产品愿景

构建一套本地优先的自主研发工作台。用户只需要输入一个较小的系统目标，系统就能通过受控的多智能体循环，持续把目标转化为可使用的软件、研究产物、报告或交互式系统。

系统不是智能体聊天室，而是一套 agent runtime。它要把模型调用转化为可验证的工作产物：计划、任务、代码补丁、实验、UI 输出、报告、测试、评审和可复用记忆。

## 2. 核心目标

给定用户的一段简短目标，平台应该能够：

1. 将目标澄清为结构化规格。
2. 根据目标、领域和使用场景推断缺失但合理的需求。
3. 当用户目标描述不足时，自动调研同类工具、常见工作流和标准功能集。
4. 判断是否需要调研、实现、UI、报告、实验，或者它们的组合。
5. 将目标拆解为可追踪、可验收的任务。
6. 将任务分配给专门角色的智能体。
7. 让智能体在隔离或受控工作区中工作。
8. 产出代码、文档、UI、报告和实验结果。
9. 通过测试、评估、截图和验收条件验证产物。
10. 自动修复失败。
11. 保留成功变更，回滚失败尝试。
12. 将有价值的知识沉淀为长期记忆。

## 3. 产品定位

本项目是一套自进化研发与软件构建工作台。

它吸收以下方向的思想：

- Agentic coding 系统，例如 Claude Code、OpenCode、OpenHands 和 SWE-agent。
- 自主实验循环，例如 autoresearch。
- 自动化科研系统。
- 本地知识库系统中的检索、记忆和结构化项目上下文。

系统可以学习公开架构思想，但不能复用泄露或专有源码。

## 4. 自迭代与自主需求扩展

系统不能表现得像“一步指令执行器”。它应该把用户的一句简短目标视为完整产品、研究或工程任务的种子。

当用户给出较笼统目标时，系统应该自主完成：

1. 深入理解目标领域，识别该类工具或系统通常应具备的能力。
2. 调研常见工具、工作流、边界场景、用户期待和质量标准。
3. 将初始目标扩展为合理产品需求集。
4. 区分必需功能、应该具备功能和可选功能。
5. 构建第一个可用版本。
6. 评估结果只是“能跑”，还是确实“有用、好用”。
7. 针对缺失能力、糟糕体验、薄弱测试或不完整文档创建后续任务。
8. 持续迭代，直到系统达到可用且相对好用的基线，或触及预算和安全边界。

示例：

如果用户说“做一个密码测试工具”，系统不应该只做一个输入框和评分。合格的 ProductAgent 和 ResearchAgent 应该发现常见工具可能包含密码强度评分、熵估算、常见弱密码检测、泄露列表检查方案、哈希示例、编码和解码辅助、随机密码生成、密码短语生成、策略检查、本地隐私保证和清晰安全提示。系统随后应根据用户场景判断哪些功能合适，实现一个连贯的功能子集，并评估最终工具是否真的可用。

## 5. 主要使用场景

### 5.1 构建小型软件工具

示例：

```text
构建一个本地 Markdown 知识库系统，支持语义搜索和问答。
```

期望输出：

- 可运行的本地应用。
- 导入和索引流程。
- 搜索和问答界面。
- 测试。
- README。
- 最终实现报告。

### 5.2 自主扩展模糊产品目标

示例：

```text
做一个密码测试工具。
```

期望输出：

- 对常见密码、安全、编码类工具进行产品调研。
- 扩展出必需功能和可选功能需求。
- 实现本地优先且隐私安全的工具。
- 在合适时提供强度评分、策略检查、生成辅助，以及部分编码或哈希工具。
- 清晰说明工具能证明什么、不能证明什么。
- 在最终报告前进行可用性评审和后续迭代。

### 5.3 调研并复现想法

示例：

```text
调研本地知识库的混合检索方法，并实现最合适的方案。
```

期望输出：

- 文献和开源项目调研。
- 结构化观点与证据。
- 实现计划。
- 实验循环。
- 基准结果。
- 保留或丢弃的补丁。

### 5.4 改进已有代码库

示例：

```text
让这个项目更容易使用，并修复明显的问题。
```

期望输出：

- 仓库扫描。
- 改进计划。
- 补丁。
- 测试。
- 评审总结。

## 6. 用户角色

### 6.1 人类所有者

提供目标、设置预算、审查关键决策，并接收最终产物。

### 6.2 智能体运行时

受控执行环境，负责管理上下文、工具、工作区、权限、记忆、日志、成本和评估。

### 6.3 专门智能体

初始角色：

- GoalSpecAgent
- PlannerAgent
- ArchitectAgent
- ProductAgent
- ResearchAgent
- CoderAgent
- UIExperienceAgent
- TesterAgent
- ReviewerAgent
- AutoCorrectionAgent
- MemoryAgent
- ReleaseAgent

## 7. 功能需求摘要

系统必须支持：

- 目标输入和 GoalSpec 生成。
- 规划和任务看板。
- 项目初始化和根指导文件。
- 命令与工作流系统。
- 上下文管理和 `/compact`。
- 人类决策交互和可配置决策颗粒度。
- 工具系统和权限控制。
- 工作区隔离或受控写入。
- 调研自动化。
- 自动实验 keep/discard 循环。
- 实现、验证、自动纠错、评审和最终报告。
- 记忆存储和检索。

## 8. 非功能需求摘要

系统必须满足：

- 本地优先。
- 模型无关，支持 OpenAI-compatible provider。
- 可观测，所有运行可审计。
- 成本可控。
- 可恢复，失败可回滚或阻塞。
- 可扩展，可增加 agent、tool、command、eval、model provider。

## 9. 成功标准

MVP 成功标准：

1. 接收自然语言目标。
2. 生成结构化目标规格和任务计划。
3. 至少能将简单欠规格目标扩展为合理验收条件。
4. 至少使用一个编码智能体编辑文件。
5. 运行验证命令。
6. 检测失败并尝试自动修复。
7. 记录每次尝试。
8. 保留成功补丁并丢弃失败补丁。
9. 生成最终报告。

完整系统成功标准：

1. 多个智能体能在隔离工作区并发运行。
2. 调研结果能转化为实现任务。
3. 能自主将模糊目标扩展为连贯产品需求和实现计划。
4. 能构建合适体验产物。
5. 跨项目维护长期记忆。
6. 不只评估最终结果，也评估执行轨迹。
7. 持续自迭代，直到产物可用、连贯且相对好用。

## 10. 非目标

早期版本不应尝试：

- 取代完整工程团队。
- 在没有策略控制的情况下运行无限制 shell。
- 在运行时稳定前构建庞大无代码平台。
- 依赖单一专有模型提供商。
- 在核心循环跑通前优先打磨漂亮 dashboard。
- 让大量智能体在没有任务、产物和评估约束下自由聊天。
- 在产物达到约定可用基线后继续无限追求完美。
